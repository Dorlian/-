# Сайт-визитка (HTML + CSS) — Git и GitHub Pages

Учебный проект: совместная разработка, ветки, Pull Request, откаты, публикация на **GitHub Pages**.

---

## Структура проекта

| Файл / папка | Назначение |
|--------------|------------|
| `index.html` | Главная страница (роль: дизайнер / архитектор) |
| `css/styles.css` | **Критичные общие стили** — при удалении ломается весь вид сайта |
| `pages/member2.html` | Персональная страница участника 2 |
| `pages/member3.html` | Персональная страница участника 3 |

## Задание 1: Правило «не работать в `main`» и методы отката

**Главное правило:** пока учитесь, опасные правки и эксперименты — только в **отдельной ветке**, не напрямую в `main`.

### Шаг 1. Инициализация репозитория и первые коммиты

```bash
git init
git branch -M main
git add .
git commit -m "feat: начальная рабочая версия сайта"
```

Дальше можно сделать ещё один «чистый» коммит (например, правка текста на главной), чтобы история была похожа на реальную.

### Шаг 2. Создать ветку для «сломанной» версии (не трогая `main` напрямую)

```bash
git checkout -b experiment/broken-version
```

### Шаг 3. Сымитировать ошибку (для демонстрации преподавателю)

- Удалить важный файл, например `css/styles.css`.
- Внести в `index.html` ошибку, которая ломает страницу (например, удалить закрывающий тег `</main>` или `</body>`).

Закоммитить это **только в ветке** `experiment/broken-version`:

```bash
git add -A
git commit -m "broken: удалён styles.css и сломана вёрстка (демо)"
```

### Шаг 4. Методы отката (что показать на 20 баллов)

Ниже — разные приёмы; на практике выберите тот, что подходит ситуации.

#### 4.1. Откатить последний коммит целиком, сохранив рабочие файлы как «до коммита»

Если плохой коммит — последний в текущей ветке:

```bash
git reset --soft HEAD~1
```

Рабочая папка останется с «плохими» изменениями; дальше можно исправить вручную или отменить файлы точечно (см. ниже).

#### 4.2. Восстановить один удалённый файл из последнего хорошего состояния ветки `main`

Не переключаясь с ветки эксперимента:

```bash
git restore --source=main -- css/styles.css
```

(В старых версиях Git: `git checkout main -- css/styles.css`.)

#### 4.3. Восстановить содержимое файла из конкретного коммита

```bash
git log --oneline
git restore --source=<hash_хорошего_коммита> -- index.html
```

#### 4.4. Полностью вернуть ветку к состоянию `main` (жёстко, осторожно)

Только если в ветке эксперимента не нужны **никакие** уникальные коммиты:

```bash
git fetch origin   # если есть remote
git reset --hard main
```

#### 4.5. Отмена уже попавшего в `main` коммита без переписывания истории (безопасно для общих репозиториев)

После того как ошибочный коммит уже в `main` и запушен:

```bash
git checkout main
git pull
git revert HEAD
git push
```

Создаётся **новый** коммит, который отменяет изменения — история линейная, удобно для командной работы.

### Шаг 5. После исправления — слияние в `main`

Когда в экспериментальной ветке всё снова работает:

```bash
git checkout main
git merge experiment/broken-version
# или через Pull Request (предпочтительно для зачёта по заданию 2)
```

## Задание 2: Совместная работа (роли и этапы)

### Роли (условные)

1. **Участник 1** — репозиторий, структура, навигация, `index.html`.
2. **Участник 2** — своя ветка и `pages/member2.html`.
3. **Участник 3** — своя ветка и `pages/member3.html`.

### Этапы (чек-лист для команды)

1. Один человек создаёт репозиторий на GitHub и добавляет остальных в **Settings → Collaborators**.
2. В `main` — базовый каркас: главная, общий `css/styles.css`, пустые или черновые страницы участников.
3. Участники 2 и 3 создают **свои ветки** (`feature/member2`, `feature/member3`), коммитят только свои страницы (и при необходимости согласованные правки CSS).
4. Каждый открывает **Pull Request** в `main`.
5. **Code review:** второй участник оставляет комментарии, автор вносит правки, повторный просмотр.
6. После апрува — **Merge** PR (лучше **Squash** или **Merge commit** — как договоритесь).
7. У всех локально: `git checkout main && git pull` — обновлённая копия.

## Публикация на GitHub Pages

1. Залейте репозиторий на GitHub (`git remote add origin ...`, `git push -u origin main`).
2. В репозитории: **Settings → Pages**.
3. **Source:** ветка `main`, папка `/ (root)` (как в этом проекте).
4. Сохраните. Через 1–3 минуты сайт будет по адресу вида  
   `https://<username>.github.io/<repo>/`.

Откройте в браузере главную: если репозиторий не в корне пользователя, путь будет с именем репозитория в URL.

---

## Что показать преподавателю (кратко)

- Ветка для опасных изменений, а не прямые коммиты в `main`.
- Как минимум два разных приёма отката: например, `git restore --source=main -- файл` и `git revert` (или `git reset` на отдельной ветке).
- Рабочий сайт на GitHub Pages по ссылке.
- Для команды: PR, комментарии review, merge, обновление через `git pull`.

## Пример истории в этом репозитории (демо отката)

После клонирования можно показать граф:

```bash
git log --oneline --graph --all
```

Ожидаемая картина: коммит «сломанной» версии и следующий за ним коммит с восстановлением файлов из `main` через `git restore --source=main`, затем слияние ветки `experiment/broken-version` в `main`.

Перед первым коммитом у себя на машине задайте имя и email Git (или только для этого репозитория):

```bash
git config user.name "Ваше Имя"
git config user.email "you@example.com"
```
